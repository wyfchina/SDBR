## 2. 仍不应伪装成完成的 UI

| 项目 | 当前状态 | 原因 | 建议 |
| --- | --- | --- | --- |
| 计划总览 `UI-OVERVIEW-001` | 未开始 | 未纳入第 1-9 验收顺序，当前仍是框架空态 | 等 `BE-BI-001` 总览聚合 API 后实施 |
| 管理后台配置编辑 | 部分入口 | `BE-UI-006` 仍是只读聚合，尚无配置持久化和变更审计 | 先做后台版本化策略中心，再开放编辑 |
| ERP/MES 连接监控 | 产品路径存在 | `BE-INT-*` 多数未开始或部分实现 | 先定义连接器契约、重试、对账和错误队列 |
| OR-Tools / Simio | 暂停 | 产品路径保留，后台集成暂停 | 项目恢复后再开放选择或验证开关 |
| 完整操作员终端 | 外部边界 | MES/SCADA 负责现场采集和设备控制 | 本系统只保留事件契约和协同看板 |

## 3. UI 后续优先级

### P0：不要继续大 UI，先补后台闭环

1. `BE-RUN-009`：计划确认、发布、撤销发布。
2. `BE-OUT-010`：面向 ERP/MES 的计划发布包和回执。
3. `BE-REL-012`：版本化 DBR 与释放策略中心。
4. `BE-UI-006`：从只读管理后台升级为受审计配置后台。

这些完成后，UI 才有明确的新页面或新动作，而不是继续做静态入口。

### P1：产品体验精修

1. 表格列宽、固定列、筛选组合和保存视图。
2. 甘特图与负荷图的可读性和细节密度。
3. 异常中心的关闭、分派、SLA 和处置历史。
4. 管理后台导入预览的行级错误、字段定位和版本差异。

这些属于现有页面加深，不改变产品边界。

### P2：分析与总览

1. `BE-BI-001`：计划总览聚合 API。
2. `BE-BI-002`：交期、吞吐、WIP、缓冲和稳定性指标。
3. `BE-BI-003/004`：流程转移、返工统计和流程拓扑挖掘。

这些能力成熟后，再实现 `UI-OVERVIEW-001`，否则总览页会变成写死指标或装饰性仪表盘。

## 4. 下一阶段建议

下一阶段建议回到后台业务闭环，而不是继续扩 UI。最自然的顺序是：

1. 计划确认、发布、撤销发布。
2. 发布包/API 契约。
3. 版本化 DBR 与释放策略中心。
4. 管理后台配置编辑。
5. 计划总览与 BI 聚合。

这样 UI 会跟着真实业务能力长出来，不会再次出现“界面和业务逻辑拧巴在一起”的问题。
