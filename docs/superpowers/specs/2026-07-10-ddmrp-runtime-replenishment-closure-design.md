# DDMRP 运行补货闭环设计

## 1. 目标与产品边界

本设计把当前 DDMRP 净流计算和只读展示扩展为 DDOM 日常运行能力，使 SDBR 能够持续回答以下问题：

- 哪些解耦点需要补货；
- 为什么需要补货、建议数量是多少；
- 应采用 Make、Buy 还是 Transfer；
- Make 建议是否具备下级物料和 CCR 能力条件；
- 计划员确认后如何形成计划制造候选、计划级容量预留和物料计划分配；
- 建议如何交给 Mock ERP 或未来 ERP，并根据回执和执行事实重新评估。

本能力属于 DDOM / S-DBR 执行层。DDAE / DDS&OP 继续负责 DDMRP 参数、Buffer Profile、调整因子、解耦点设计和主设置审批。SDBR 只执行“接收、校验、冻结、运行、反馈”，不得在页面中重新计算、配置或审批 DDAE 主参数。

ERP/WMS/QMS 继续作为 BOM、库存账、正式采购/制造/调拨订单、批次和质量状态的权威系统。SDBR 只建立计划建议、计划制造候选、计划级容量预留和计划级物料分配，不宣称这些对象是外部系统的正式订单或库存账占用。

本设计扩展现有 `BE-DDMRP-001` 至 `BE-DDMRP-006` 的运行闭环。实施前必须先更新 `docs/backend-specification.md` 和 `docs/ui-specification.md`，增加版本化评估、建议治理、BOM 预览、计划预留、外部投递和工作台验收条件。

## 2. 已确认业务原则

1. DDMRP 判断以 `docs/ddom-ddmrp-runtime-principles.md` 为准：红区、黄区补货；绿区和高于绿区不补货。
2. 计划状态由净流位置判断，执行状态由在手可用量单独判断，两者不得混用。
3. 所有 Make、Buy、Transfer 建议均由计划员确认，系统不得自动建立正式订单。
4. Make 建议由 SDBR 做执行级多级 BOM 影子展开、物料可用性判断和 CCR 负荷预览；ERP 建立正式制造、采购或调拨订单。
5. BOM 由 ERP 管理，SDBR 消费并冻结 Plan BOM、`BomVersionID`、生效时间和来源。
6. BOM 展开遇到 DDMRP 解耦点时停止，由该解耦点的净流结果决定补货；非解耦自制件继续展开。
7. 多个上层需求使用相同物料地点时，按“物料 + 地点 + 需求时间窗”汇总一次净流计算，并保留需求来源明细。
8. 计划员确认 Make 建议后，立即以同一事务建立计划制造候选、CCR 计划级容量预留和下级物料计划分配。
9. 计划级容量和物料预留立即影响后续评估，但只有 ERP 接受并返回正式订单后才转为正式占用。
10. ERP 拒绝、确认超时、建议撤销或订单取消时，释放相应计划级预留并保留审计记录。
11. 下级物料不足不阻止计划员确认顶层补货需求，但计划制造候选必须标记为“待物料满足”，不得释放；系统同时生成下级 Make、Buy 或 Transfer 建议。
12. 每次 DDMRP 评估不可变。未确认建议可被新评估替代；已确认建议不得被静默修改。
13. 系统采用事件触发的局部重评估和每班/每日全量核对相结合的方式。
14. SDBR 不因生成补货建议而自动重排现有正式计划。

## 3. 权威输入与冻结

### 3.1 DDAE 批准配置

SDBR 按契约消费并冻结：

- `OperatingModelConfigurationID` 和 fingerprint；
- `DDMRPConfigurationID`；
- 解耦点；
- Stock Buffer Profile 结果与红黄绿边界；
- DLT、MOQ、订货倍数和已批准调整结果；
- 需求尖峰资格和在途资格规则；
- 资源角色、控制点和与 Make 预览相关的批准排程策略。

SDBR 不解释契约之外的隐式含义，也不在本地补充缺失的 DDAE 主参数。契约不足时必须提交 Contract Agent 变更请求。

### 3.2 ERP/WMS/QMS 运行事实

运行评估至少消费：

- 物料、地点和 UOM；
- 在手库存、已分配库存和质量可用状态；
- 在途供应、预计到达时间和供应状态；
- 客户需求、制造需求、补货需求和需求时间；
- Plan BOM、BOM 版本、组件用量和 Make/Buy/Transfer 来源类型；
- 正式订单、收货、领料、完工和质量放行事实。

待检、冻结、报废或未放行库存不得计入合格可用量。已分配库存不得重复承诺。在途供应是否合格必须遵循冻结的检查窗口和状态规则。

### 3.3 SDBR 执行事实

- 已确认但未转正式订单的计划制造候选；
- CCR 计划级容量预留；
- 下级物料计划分配；
- 已存在的正式排程和资源负荷；
- SDBR 资源日历和有效产能窗口；
- 尚未由 ERP 回执完成的建议包与投递状态。

### 3.4 评估冻结

每次评估冻结：

- 评估时间和触发原因；
- DDAE 配置版本和 fingerprint；
- BOM 版本；
- 库存、需求、供应、质量状态和计划分配快照；
- CCR 日历、正式负荷和计划级预留；
- 数据来源、时间戳和 trace ID。

历史评估不受后续配置、BOM、库存或供应变化影响。

## 4. 核心领域组件

### 4.1 DDMRP Evaluation Run

代表一次不可变评估。负责记录评估范围、冻结输入、结果摘要、结构化问题、前序版本和替代关系。运行可以是全量评估，也可以只覆盖受事件影响的物料地点。

### 4.2 Net Flow Engine

负责合格需求、合格供应、净流位置、计划状态、执行状态和建议数量计算。计算引擎只消费冻结参数，不负责参数治理。

### 4.3 BOM Explosion Preview

负责 Make 建议的 Plan BOM 影子展开。它沿非解耦自制件继续展开，在解耦点停止，并把 Buy/Transfer 组件转换为对应建议候选。它必须检测 BOM 循环、缺失 UOM、非法用量、版本失效和无法解析的物料地点。

### 4.4 Replenishment Recommendation

保存 Make、Buy、Transfer 建议及其数量、目标时间、可行时间、风险、来源需求和治理状态。建议与产生它的评估行、配置版本和 BOM 版本绑定。

### 4.5 Feasibility Preview

保存计划员确认前的业务证据：BOM 展开、下级短缺、物料可用性、CCR 负荷前后变化、保护线状态、标准目标到货时间、最早可行时间和延迟风险。确认时冻结该预览。

### 4.6 Planning Reservation

由以下两个独立台账组成：

- `CCRCapacityReservation`：日期/班次或有效产能窗口上的计划级容量预留；
- `MaterialPlanningAllocation`：防止库存被多个候选重复承诺的计划级物料分配。

两类预留必须与计划制造候选在同一事务中建立，任何一步失败都必须全部回滚。

### 4.7 ERP Adapter 与投递台账

连接层使用稳定 canonical 建议包。第一版由 Mock ERP 接收；未来 Direct Adapter 或 UNS 只替换传输方式，不改变建议语义。投递台账保存幂等键、尝试次数、ACK、正式订单引用、重放、失败和死信状态。

### 4.8 Event Reevaluation

负责把订单、库存、收货、领料、质量放行、在途、分配、正式订单回传和 DDAE 配置激活事件映射为受影响的物料地点范围，并触发新评估。周期全量核对用于发现漏事件和数据漂移。

## 5. 核心计算规则

### 5.1 净流位置

```text
NetFlowPosition = QualifiedOnHandQty
                + QualifiedOpenSupplyQty
                - QualifiedDemandQty
```

合格需求、需求尖峰和合格在途的定义来自冻结的 DDAE 配置。SDBR 不创建本地替代口径。

### 5.2 状态与补货量

- 净流位置决定 Red、Yellow、Green 或 AboveGreen 计划状态；
- 在手合格库存单独决定执行状态；
- Red 和 Yellow 才生成补货建议；
- Green 和 AboveGreen 的建议补货量必须为零；
- 补货目标为 TopOfGreen，并按冻结的 MOQ 和订货倍数修正。

### 5.3 需求汇总与追溯

同一物料地点在同一需求时间窗内统一计算，避免按每个上层订单重复补货。汇总行必须保留每个订单、建议、BOM 父项、需求数量、需求时间和 trace ID。已确认计划分配必须计入占用。

### 5.4 BOM 展开

1. 使用评估冻结的 Plan BOM；
2. 按父项数量、组件用量和 UOM 转换计算下级需求；
3. 遇到 DDMRP 解耦点后停止展开并触发该物料地点的净流评估；
4. 非解耦 Make 项继续展开；
5. Buy/Transfer 项形成外部建议候选；
6. 缺少 BOM、UOM、数量、版本或来源类型时，禁止生成可执行候选；
7. BOM 循环必须终止并形成结构化问题。

### 5.5 时间语义

- `EvaluationAt`：本次 DDMRP 运行采用运行事实快照的时间点，不是客户交期、正式释放时间或实际开工时间；
- `StandardTargetReceiptAt`：仅在契约明确 DLT 的时间单位、工作日历和目标日期语义时，以 `EvaluationAt + FrozenDLT` 形成标准补货到位目标；它不是生产承诺；
- `EarliestFeasibleCompletionAt`：Make 建议经过 BOM、物料、CCR 和日历影子试排后得到的最早可行完工时间；
- Buy/Transfer 使用标准目标到货时间作为需求日期；
- Make 同时保留标准目标到货时间和最早可行完工时间；
- 最早可行时间晚于标准目标时，显示补货响应延迟风险。

若计划员确认前输入版本已变化，旧预览必须失效并重新评估。

### 5.6 CCR 影响预览

Make 建议确认前读取当前正式负荷、已有容量预留、SDBR 资源日历和冻结保护线，计算新增负荷前后状态。预览不得修改正式排程。确认后才建立计划级容量预留。

## 6. 建议治理与状态

### 6.1 主状态流

```text
NewSuggestion
  -> PendingReview
  -> Confirmed
  -> Issued
  -> ERPAccepted
  -> InExecution
  -> Completed
```

业务界面显示“新建议、待审核、已确认、已输出、ERP 已接受、执行中、已完成”。

### 6.2 旁路状态

- `Rejected`：计划员拒绝；
- `Withdrawn`：尚未转正式订单前主动撤销；
- `Superseded`：未确认建议被新评估替代；
- `OutputFailed`：投递失败，可重试；
- `AdjustmentRequired`：已确认或已输出建议因事实变化需要调整。

### 6.3 不可变与变更规则

- 未确认建议可由新评估自动替代；
- 已确认但未输出建议若数量或日期变化，必须重新确认；
- 已输出建议只能生成加急、延期、变更或取消提议，不得静默覆盖；
- 所有替代、拒绝、撤销和调整必须保留版本链和审计事件。

## 7. Make 确认与计划预留

计划员确认 Make 建议时，系统必须在同一事务中：

1. 校验建议、评估和预览仍为最新；
2. 再次检查物料和 CCR 并发占用；
3. 建立 `PlannedManufacturingCandidate`；
4. 建立 CCR 计划级容量预留；
5. 建立下级物料计划分配；
6. 冻结评估、BOM、配置、快照和预览引用；
7. 记录计划员、确认时间和审计原因。

若下级物料不足：

- 允许确认顶层补货需求；
- 计划制造候选标记为“待物料满足”；
- 不允许进入工单释放；
- 自动形成下级 Make、Buy 或 Transfer 待审核建议；
- 根据预计下级到位时间重新计算 Make 建议完工时间和容量预留窗口。

计划预留立即参与后续净流、物料可用性和 CCR Planned Load 判断。ERP 接受后，计划制造候选和预留关联正式订单；ERP 拒绝、超时、撤销或取消时释放预留并保留记录。

## 8. 外部建议包与反馈闭环

建议包至少包含：

- 建议 ID、版本和幂等键；
- Make/Buy/Transfer 类型；
- 物料、地点、UOM、数量；
- 标准目标到货时间和最早可行时间；
- DDMRP 评估、配置、BOM 和快照引用；
- 需求来源摘要；
- Make 的计划制造候选、下级物料和 CCR 预览摘要；
- 风险、限制和 trace ID。

第一版 Mock ERP 返回 Accepted、Rejected 或 Failed ACK，并可在 Accepted 时返回正式订单引用。重复消息不得重复创建订单。传输失败可重试，超过投递策略进入死信并保留计划预留，等待计划员处理；已经成功送达但超过业务 ACK 时限仍未被 ERP 接受的建议，按冻结的超时策略释放预留。外部连接器不得修改 DDMRP 核心业务语义。

正式订单、收货、领料、完工和质量放行事件返回后，更新对应运行事实并触发新评估。历史建议和评估保持不变。

## 9. 重评估与数据新鲜度

### 9.1 事件触发

- 客户订单、计划订单或补货需求新增、变更、取消；
- 库存收货、发料、退料、调整和质量放行；
- 在途数量、日期或状态变化；
- 计划分配建立、释放或调整；
- ERP 正式订单回传；
- DDAE 新配置版本激活。

事件只重算受影响的物料地点及其需求链。

### 9.2 周期核对

每班或每日执行一次全量重算，检查漏事件、版本不一致和数据漂移。周期由运行作业配置管理，不在 DDMRP 参数页面治理。

### 9.3 新鲜度门控

计划员确认前，系统必须检查库存、供应、需求、BOM、配置和 CCR 版本。任一关键输入变化时，拒绝使用旧预览并要求重新评估。

## 10. 工作台设计

现有“物料计划”页面升级为 DDMRP 日常运行工作台。它不提供 DDMRP 参数配置。

### 10.1 顶部摘要

- 最新评估时间和数据新鲜度；
- Red、Yellow、Green、AboveGreen 数量；
- 待确认、待物料、已输出和输出失败数量；
- Make、Buy、Transfer 建议数量；
- 缺失关键数据的物料数量。

### 10.2 主列表

显示物料、地点、计划状态、执行状态、缓冲百分比、在手可用量、已分配量、有效在途、合格需求、净流位置、建议数量、来源类型、目标时间和风险。支持搜索、筛选、排序和分页。

### 10.3 建议详情

业务详情显示：

- 需求与供应构成；
- 缓冲区判断依据；
- 标准目标到货时间和最早可行时间；
- Make 的 BOM 展开、下级短缺、CCR 负荷前后变化；
- 推荐动作和阻塞原因；
- 建议状态和外部回执。

配置 ID、fingerprint、幂等键、原始错误码和接口路径放入可展开技术详情。

### 10.4 操作

- 重新评估；
- 确认建议；
- 拒绝建议；
- 撤销尚未输出的建议；
- 输出到 Mock ERP；
- 重试失败投递；
- 查看版本、预留和 ACK 历史。

MTA 库存缓冲建议继续进入统一缓冲优先级视图，并明确显示来源为 MTA；不得与 MTO 时间缓冲混成无法解释的颜色数量。

## 11. API 边界

API 按现有 `/planner/workbench/ddmrp` 路径增量扩展：

- 评估运行：创建、查询、查看结果和问题；
- 建议：查询、预览、确认、拒绝、撤销；
- 输出：生成建议包、Mock 投递、重试和查询 ACK；
- 预留：只读查询计划制造候选、CCR 容量预留和物料计划分配；
- 事件：由内部集成事件触发局部重评估；
- 历史：查询评估版本、建议替代关系和审计事件。

第一版不新增 DDMRP 参数编辑、Buffer Profile 治理或 DDS&OP 场景入口。

## 12. 异常与并发处理

- DDAE 配置未批准、fingerprint 不一致或全局配置不可用：整次评估拒绝；
- 单个物料缺少 BOM、UOM、地点或来源类型：该物料标记为数据不完整，不生成可执行建议，其他物料可继续；
- BOM 循环、数量非法或版本失效：停止受影响需求链并返回具体节点；
- 确认时版本已变化：拒绝旧确认并要求重新评估；
- 物料或容量并发冲突：确认事务全部回滚；
- ERP 传输失败：保留已确认建议和计划预留，进入可重试状态；
- 传输重试超过策略：进入死信并保留预留，等待计划员处理，不重复创建候选或订单；
- 已成功送达后 ERP 拒绝、业务 ACK 超时或建议撤销：释放计划级预留并保留审计。

同一评估请求、确认动作和外部投递必须幂等。服务重启、worker 重试或消息重放不能产生重复建议、重复候选、重复预留或重复外部订单。

## 13. 分阶段实施

### 阶段 1：版本化评估与运行工作台

- 拆分现有 DDMRP 计算服务；
- 建立不可变评估运行、评估行和问题；
- 支持事件范围与全量评估；
- 升级只读工作台和数据新鲜度展示。

### 阶段 2：建议生命周期与计划员确认

- 建立 Make、Buy、Transfer 建议；
- 实现待审核、确认、拒绝、替代和历史；
- 增加确认前版本门控和幂等控制。

### 阶段 3：Make 可行性预览与计划预留

- 多级 BOM 影子展开；
- 下级物料可用性和补货级联；
- CCR 影子负荷预览；
- 计划制造候选、CCR 容量预留和物料计划分配。

### 阶段 4：Mock ERP 与事件反馈闭环

- canonical 建议包；
- Mock ERP 投递、ACK、重试和死信；
- 正式订单引用和预留转正/释放；
- 执行事件触发的局部重评估与周期全量核对。

每个阶段必须独立可验收，不得用后续阶段尚未实现的接口伪装闭环完成。

## 14. 验收测试范围

1. 净流公式、Red/Yellow/Green/AboveGreen 边界正确。
2. Green 和 AboveGreen 的建议补货量始终为零。
3. 计划状态与执行状态使用不同输入。
4. 已分配库存不被重复使用。
5. 在途时间窗、质量状态和需求尖峰按冻结配置判断。
6. 同一物料地点的多个需求只形成一次汇总净流，但来源明细完整。
7. 补货量按 TopOfGreen、MOQ 和订货倍数修正。
8. BOM 在解耦点停止，非解耦 Make 继续展开，Buy/Transfer 生成对应建议。
9. BOM 循环、缺失 UOM、非法用量和无法解析引用阻止可执行候选。
10. 标准目标到货时间和最早可行完工时间语义分离。
11. Make 预览展示下级短缺和 CCR 负荷前后变化。
12. 下级不足时允许确认顶层需求，但候选不可释放并生成下级建议。
13. 确认 Make 建议时，候选、容量预留和物料分配原子建立。
14. 重复确认不生成重复候选或预留。
15. 输入版本变化后，旧预览不能确认。
16. ERP Accepted、Rejected、Failed、重试和死信状态正确。
17. ERP 拒绝、超时、撤销时按规则释放计划预留。
18. 事件触发局部重评估，周期作业完成全量核对。
19. 历史评估和已确认建议不可变，替代关系可追溯。
20. 工作台使用中文业务表达，技术信息默认折叠。
21. 现有排程、释放、MTO/MTA、DDMRP 净流和 API 测试不回归。

## 15. 非目标

- 不在 SDBR 配置、审批或重算 DDAE DDMRP 主参数；
- 不实现 Buffer Profile 治理、调整因子审批或 DDS&OP 场景管理；
- 不自动确认补货建议；
- 不直接建立 ERP 权威采购单、制造工单或调拨单；
- 不修改 ERP/WMS 库存账或 QMS 质量状态；
- 不实现替代料、批次/效期、合批或复杂拆批规则；
- 不因补货建议自动重排正式生产计划；
- 不把影子 BOM、物料或 CCR 预览称为正式生产可行性承诺；
- 不以 Simio 替代日常 DDMRP 净流和补货建议计算。
