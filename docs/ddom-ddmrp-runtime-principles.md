## 1. DDOM 与 DDMRP 的定位

DDOM 是运营执行模型，目标是在日常/实时范围内保护和促进信息与物料流动。资料中 DDOM 的核心结构包括战略解耦点、战略控制点、动态缓冲，以及按实际需求驱动的节拍运行。

DDMRP 是 DDOM 中用于战略解耦点的供应订单生成与库存缓冲执行机制。它不等同于 DDS&OP，也不负责配置模型；它消费已经生效的主设置、解耦点、库存缓冲区和需求/供应事实，输出补货建议、相对优先级和运行状态。

本系统原则：

- DDS&OP 或外部主设置负责决定“模型应该如何配置”。
- DDOM/SDBR 负责决定“今天如何按已生效配置运行”。
- 本系统接受配置，不提供 DDMRP 参数治理页面。
- 本系统向 DDS&OP 或管理层回传运行偏差：库存表现、缓冲表现、产能冲突、执行异常、流动速度和计划/实际偏差。

## 2. DDMRP 运行所需参数

这些参数在本系统中应作为外部已配置输入或运行事实，不应在 DDOM 日常页面中重新设计。

| 参数/数据 | 含义 | 本系统使用方式 | 配置归属 |
| --- | --- | --- | --- |
| 解耦点 | 需要通过库存缓冲保护的位置，通常是物料-地点组合 | 冻结到 Master Data Version，用于识别需要计算净流的位置 | DDS&OP/外部主设置 |
| Buffer Profile | 物料类型、提前期、波动等配置组合 | 第一版只记录 profile ID，不计算其参数 | DDS&OP |
| DLT | 解耦提前期，用于缓冲区大小、尖峰窗口和订单请求日期 | 作为输入，第一版用于需求/供应检查窗口和说明 | DDS&OP/主设置 |
| 红区/黄区/绿区数量 | 库存缓冲的分区结果 | 直接消费区间数值，计算状态和建议量 | DDS&OP/主设置 |
| Top of Red | 红区顶部 | 用于库存执行状态与告警 | 派生自红区数量 |
| Top of Yellow | 红区 + 黄区 | 用于判断是否触发补货 | 派生自红黄区数量 |
| Top of Green | 红区 + 黄区 + 绿区 | 用于计算补货目标量 | 派生自红黄绿区数量 |
| 在手库存 | 当前物理可用库存 | 参与净流计算，并单独判断执行状态 | ERP/库存系统权威 |
| 在途/未收货供应 | 已下达但未接收的采购、转储或制造供应 | 参与净流计算；第一版可按有效窗口过滤 | ERP/供应订单权威 |
| 合格需求 | 逾期需求、今日需求、合格尖峰需求 | 从需求信号中筛选后扣减净流 | ERP/订单或需求系统 |
| 订单尖峰阈值 | 判断未来某日需求是否构成尖峰 | 第一版可直接接收 `IsQualifiedSpike`，不配置阈值 | DDS&OP/主设置 |
| 订单尖峰窗口 | 判断尖峰的未来时间范围，通常至少覆盖 DLT | 第一版可直接接收合格尖峰标记或以 DLT 窗口辅助判断 | DDS&OP/主设置 |
| MOQ/订单倍量 | 最小订货量和订单倍数 | 修正补货建议量 | ERP/主数据 |

## 3. 净流位置规则

DDMRP 计划计算使用净流位置，而不是传统按预测逐层净到零。

```text
Net Flow Position = On Hand + On Order - Qualified Demand
```

在本系统中对应为：

```text
净流位置 = 在手库存 + 有效在途供应 - 合格需求
```

合格需求包括：

- 逾期未满足需求。
- 今日到期需求。
- 在订单尖峰窗口内达到尖峰阈值的未来需求。

原则：

- 普通未来需求不直接进入净流扣减，因为它已经通过缓冲区设计被吸收。
- 预测不直接驱动日常补货订单；预测可以作为外部缓冲参数计算的输入，但不在本系统第一版中配置。
- 尖峰需求一旦合格，按当天累计需求全量计入，而不是只计入超过阈值的部分。
- 尖峰资格应在每次净流计算时重新判断，避免生成供应后又错误显示为过量。

## 4. 补货建议规则

核心触发原则：

```text
当净流位置 <= Top of Yellow 时，触发补货建议。
当净流位置 > Top of Yellow 时，不触发补货建议。
```

补货建议量：

```text
建议补货量 = Top of Green - Net Flow Position
```

随后按 MOQ 和订单倍量修正。

状态解释：

| 净流位置 | 计划状态 | 是否补货 | 业务含义 |
| --- | --- | --- | --- |
| `<= Top of Red` | 红区 | 是 | 需要优先处理，库存缓冲保护能力受到明显威胁 |
| `> Top of Red` 且 `<= Top of Yellow` | 黄区 | 是 | 正常补货触发区，应生成供应建议补到绿区顶部 |
| `> Top of Yellow` 且 `<= Top of Green` | 绿区 | 否 | 缓冲仍在计划保护范围内，继续监控 |
| `> Top of Green` | 高于绿区 | 否 | 可能偏高，通常不建议新增供应 |

重要更正：

- 绿区不代表“补到绿区顶部”的触发区。
- 黄区才是常规补货区。
- 红区是更高优先级的补货和干预区。

## 5. 计划状态与执行状态必须分离

资料明确区分计划视图和执行视图：

- 计划状态基于净流位置，用于判断是否生成供应订单。
- 执行状态基于在手库存或 projected on-hand，用于管理已经打开的供应订单、库存告警和同步风险。

本系统原则：

- `PlanningStatus` 使用净流位置。
- `ExecutionStatus` 使用在手库存。
- 不得用净流位置替代在手库存告警。
- 不得用在手库存直接决定是否生成补货建议。

## 6. 相对优先级规则

DDMRP 的优先级不是单纯按交期排序，而是通过颜色和百分比形成相对优先级。

计划优先级：

```text
Planning Priority % = Net Flow Position / Top of Green
```

排序原则：

1. 先按颜色排序：红区优先于黄区，黄区优先于绿区。
2. 同色内按百分比排序，百分比越低，优先级越高。
3. 高于绿区通常不进入补货行动列表。

执行优先级：

- 对库存缓冲，优先关注在手库存红区/黄区穿透。
- 对工单和控制点，优先关注时间缓冲颜色、渗透率、到达状态和异常。
- 交期仍是解释字段，但不应成为唯一优先级来源。

## 7. 与排程、释放和 MES 派工的关系

DDMRP 产生的是供应/制造建议和库存缓冲优先级；Demand Driven Scheduling 对制造订单进行有限排程；Demand Driven Execution 管理库存、时间和产能缓冲。

本系统关系如下：

| 模块 | 消费 DDMRP 的方式 | 不应做的事 |
| --- | --- | --- |
| Planning Run | 可使用 DDMRP 输出的制造建议作为待排工单来源 | 不在求解器中重新设计 Buffer Profile |
| CP-SAT 排程 | 按资源、日历、路线、控制点约束排制造订单 | 不把 DDMRP 净流直接硬编码为资源约束 |
| 释放门控 | 使用库存风险、物料可用和缓冲状态解释是否释放 | 不把绿色库存误判为必须补货 |
| 缓冲执行 | 按颜色和渗透率形成现场优先级 | 不只按客户交期派工 |
| MES 派工建议 | 输出已释放工序的建议顺序 | 不绕过释放门控直接派工 |
| DDS&OP 反馈 | 回传缓冲穿透、补货频率、缺料、延误和产能冲突 | 不在 DDOM 页面做战术配置审批 |

## 8. 当前产品第一阶段应实现的能力边界

应实现：

- 接收解耦点、缓冲区、需求、在途供应和库存事实。
- 按净流方程计算 DDMRP 计划状态。
- 红/黄区生成补货建议，绿区和高于绿区不补货。
- 区分计划状态和执行状态。
- 提供只读 DDMRP 运行状态。
- 将库存缓冲风险作为释放、派工和偏差分析的解释数据。

不应实现：

- DDMRP 参数配置 UI。
- Buffer Profile 设计或治理。
- 调整因子审批。
- DDS&OP 会议、场景评估或模型重构流程。
- 预测驱动的日常供应订单生成。
- 将所有物料净到零的传统 MRP 行为。

## 9. 当前系统需保持的测试原则

- 红区、黄区、绿区、高于绿区都要有可重复测试案例。
- 绿区案例必须验证“不补货”。
- 黄区案例必须验证“补到 Top of Green”。
- 在手库存执行状态必须能和净流计划状态不同。
- DDMRP 测试数据必须使用测试前缀，不混入正式数据。
- UI 只显示运行结果和业务解释，不出现参数配置入口。
