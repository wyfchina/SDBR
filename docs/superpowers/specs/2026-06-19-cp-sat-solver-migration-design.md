# CP-SAT 求解器迁移设计

日期：2026-06-19

## 1. 决策

当前唯一可用于新 Planning Run 和重排执行的求解器切换为 OR-Tools CP-SAT，内部标识继续使用 `ortools`。Gurobi 进入暂停状态：保留实现、历史运行记录和历史排程结果读取能力，但不得用于创建或执行新的计划。

本设计采用“统一契约下逐项等价迁移”，不重写 Planning Run、计划输出、释放门控或发布生命周期。

## 2. 目标

- CP-SAT 实现现有 Gurobi 模型已经验证的全部基础约束与目标。
- 上层继续消费 `SchedulingProblem`、`SchedulingResult` 和结构化诊断。
- 测试数据通过 CP-SAT 完成 Planning Run、排程输出、释放门控和发布生命周期闭环。
- UI 中 CP-SAT 是唯一可选引擎，Gurobi 显示“已暂停”。
- 后续高级排程能力只在 CP-SAT 路径上开发。

## 3. 非目标

- 本阶段不删除 Gurobi 代码、依赖探测或历史数据。
- 不保证 CP-SAT 与 Gurobi 在存在多个同等最优解时生成完全相同的工序顺序。
- 不在本阶段实现 Simio。
- 不把物料 MRP、WIP 门控或发布治理合并进求解器。
- 不在迁移阶段同时实现换型矩阵、冻结区、批次和完整多目标策略；这些在等价迁移后按原高级排程路线继续开发。
- `release_not_before` 当前只有输入字段，现有 Gurobi 模型没有施加该约束；它归入迁移后的高级 CP-SAT 约束，不作为等价迁移验收项。

## 4. 架构边界

`Planning Run -> SchedulingProblem -> Solver Adapter -> SchedulingResult -> 计划输出/释放门控/发布生命周期`

求解器适配层承担：

- 输入校验与分钟级整数时间转换。
- CP-SAT 变量和约束构建。
- 求解时间限制和状态映射。
- 赋值结果及结构化诊断生成。

上层业务模块不直接引用 `cp_model`，也不感知 CP-SAT 内部变量。

## 5. 约束等价矩阵

| 现有能力 | CP-SAT 表达 | 验收重点 |
| --- | --- | --- |
| 工序持续时间 | 整数开始、结束和 IntervalVar | 结束时间等于开始时间加持续时间 |
| 单一或备用资源 | 每个候选资源一个可选 IntervalVar；presence 之和为 1 | 只能选择一个资源，并返回实际资源 |
| 约束资源有限产能 | `AddNoOverlap` | 同一有限资源上的已选工序不重叠 |
| 非约束资源无限产能 | 不增加资源互斥约束 | 保持冲刺能力语义 |
| 工序优先和最小间隔 | `start_after >= end_before + lag` | 路线顺序与最小间隔成立 |
| 日历/能力桶 | 工序选择合法能力窗口；桶内总时长受限 | 工序不跨越不可用窗口，容量不足时不可行 |
| 保护交期与迟期 | 订单完工最大值、迟期非负变量 | 优先使用保护交期 |
| 完工期 | 所有工序结束时间最大值 | 与现有目标语义一致 |
| 备用资源惩罚 | presence 加权求和 | 无排程收益时优先主资源 |
| 时间限制 | `max_time_in_seconds` | 区分最优、可行、超时无解和不可行 |

CP-SAT 使用整数目标系数。现有浮点权重先按固定比例转换为整数，并在构模前校验非负性与缩放范围；对外仍返回统一的目标值和诊断。

## 6. 状态和诊断映射

| CP-SAT 状态 | 统一状态 |
| --- | --- |
| `OPTIMAL` | `Optimal` |
| `FEASIBLE` | `Feasible` |
| `INFEASIBLE` | `Infeasible` |
| `UNKNOWN` 且达到时间限制 | `TimeLimit` |
| `MODEL_INVALID` | `Error` |

诊断代码使用 `ORTOOLS_*` 前缀。已有与引擎无关的输入错误代码，如 `MISSING_SCHEDULE_START` 和 `DUPLICATE_OPERATION_ID`，保持不变。

## 7. 切换和兼容

- `create_solver_engine("ortools")` 默认探测 OR-Tools 安装并返回可执行引擎。
- `create_solver_engine("gurobi")` 仍可用于读取能力信息和专门的历史兼容测试，但产品可用性返回暂停，业务 API 拒绝新执行。
- 新建 Planning Run、重排执行和测试数据默认使用 `ortools`。
- 已完成的 Gurobi Run 不迁移、不重算，计划结果和发布包继续可查询。
- 旧的 Pending/Queued Gurobi Run 不自动执行；对外返回“求解器已暂停”，由计划员复制为新的 CP-SAT Run。

## 8. UI 语义

- 创建排程向导：`OR-Tools CP-SAT` 可选且默认选中。
- Gurobi：同级显示但禁用，状态为“已暂停”。
- Simio：继续显示“暂不可用”。
- 排程结果保留历史求解器名称，不能把旧 Gurobi 结果显示成 CP-SAT。
- 计划发布治理采用排程结果内嵌方案；生命周期标签显示“草案、已复核、已批准、已发布、已撤销、已被替代”，内部枚举保持兼容。

## 9. 实施顺序

1. 修改后端规格状态和验收条件：恢复 `BE-SOLVER-009`，将 `BE-SOLVER-002` 标记为暂停，并更新运行与重排的活动引擎边界。
2. 为 CP-SAT 约束等价矩阵补充高价值契约测试，先验证测试因未实现而失败。
3. 实现 CP-SAT 模型、状态映射和结构化诊断。
4. 将 Planning Run、重排和测试数据切换到 CP-SAT，并拒绝新的 Gurobi 执行。
5. 验证三组 `TST` 场景的计划输出和释放门控。
6. 修改 UI Spec，再将创建向导和管理后台能力状态切换为 CP-SAT 可用、Gurobi 暂停。
7. 在求解器迁移验收后实施排程结果内嵌的计划发布治理 UI。

## 10. 测试策略

高价值测试覆盖：

- 缺少计划起点、重复工序编号等输入错误。
- 固定资源有限产能与工序不重叠。
- 工序优先及最小间隔。
- 备用资源选择和主资源惩罚。
- 能力桶合法窗口与不可行场景。
- 保护交期优先级、迟期和完工期目标。
- 时间限制与 CP-SAT 状态映射。
- Planning Run、重排、输出、释放和发布闭环使用 CP-SAT。
- 新 Gurobi 运行被拒绝，历史 Gurobi 结果仍可读取。
- UI 能力状态、默认选择和中英文标签。

相关测试通过后运行完整 `pytest -q` 回归。性能测试记录工单数、工序数、候选资源数、时间限制、状态、目标值和运行时间，不以 Gurobi 排程顺序完全一致作为验收条件。

## 11. 风险控制

- 能力桶表达是迁移中风险最高的部分，必须单独验证窗口边界、跨日班次和容量不足。
- CP-SAT 仅接受整数变量，时间统一为分钟，目标权重统一缩放，禁止隐式浮点截断。
- 多个同等最优解可能产生不同顺序；业务验收比较可行性、约束满足和 KPI，而非逐行完全一致。
- 暂停 Gurobi 只限制新运行，不破坏审计链和历史计划可追溯性。
